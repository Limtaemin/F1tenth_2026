# 00. F1TENTH Race Stack 전체 개요

이 문서는 이 레포가 어떤 구조로 되어 있고, 각 패키지가 무슨 역할을 하는지 설명한다.

---

## 1. 이 레포의 목적

이 레포는 F1TENTH 차량을 ROS2 환경에서 구동하기 위한 실험용 자율주행 스택이다.

주요 목표는 다음과 같다.

```text
1. 차량 하드웨어 bringup
2. LiDAR, VESC, odom, tf 확인
3. 웹/키보드/조이스틱 수동 조작
4. SLAM으로 맵 생성
5. 저장된 맵에서 위치추정
6. waypoint/raceline 기반 경로추종
7. LiDAR 기반 장애물 회피
8. safety node를 통한 충돌 방지
2. 전체 시스템 흐름

기본 구조는 다음과 같다.

LiDAR /scan
        │
        ├── SLAM Toolbox → /map 생성
        │
        ├── Particle Filter → 현재 위치 추정
        │
        └── 장애물 회피 알고리즘

VESC / sensors
        │
        └── /odom 생성

자율주행 알고리즘
        │
        ├── Pure Pursuit
        ├── Stanley Avoidance
        └── Gap Follower
                │
                ▼
             /drive

수동 조작
        │
        ├── Joystick → /teleop
        ├── Web Dashboard → /webop
        └── Keyboard Teleop → /drive 또는 /webop
3. 명령 토픽 구조

F1TENTH에서 차량을 움직이는 명령은 보통 Ackermann 메시지로 전달된다.

주요 토픽은 다음과 같다.

토픽의미
/drive자율주행 알고리즘 또는 직접 제어 명령
/teleop조이스틱 수동 조작 명령
/webop웹 대시보드 수동 조작 명령
/scanLiDAR 거리 데이터
/odom차량 odometry
/mapSLAM 또는 저장된 occupancy map
/tf좌표계 변환 정보
4. mux의 역할

여러 곳에서 주행 명령이 동시에 나올 수 있다.

예를 들면 다음과 같다.

자율주행 코드 → /drive
조이스틱     → /teleop
웹 대시보드 → /webop

이때 ackermann_mux가 어떤 명령을 최종적으로 차량에 보낼지 결정한다.

일반적인 우선순위는 다음과 같다.

웹 대시보드 /webop
> 조이스틱 /teleop
> 자율주행 /drive

따라서 웹 대시보드는 자율주행 실험 중 비상 개입용으로 사용할 수 있다.

5. 주요 패키지 설명
5.1 f1tenth_system

차량 하드웨어 bringup을 담당한다.

포함 기능:

VESC driver
Ackermann to VESC 변환
VESC odometry
LiDAR driver
joystick teleop
ackermann mux
sensor 설정
5.2 mission_control

실험자가 차량을 쉽게 조작하기 위한 도구들이 들어 있다.

주요 파일:

파일역할
dashboard.py웹 대시보드
key_teleop.py터미널 키보드 조작
odom_tuner.py고정 속도/조향각으로 odom 튜닝
5.3 particle_filter

저장된 맵에서 차량의 현재 위치를 추정한다.

즉, SLAM이 맵을 만드는 단계라면, particle filter는 이미 만들어진 맵 안에서 차량이 어디 있는지 찾는 단계다.

5.4 pure_pursuit

CSV raceline을 따라가는 경로추종 알고리즘이다.

기본 흐름:

현재 위치 확인
→ 앞쪽 lookahead waypoint 선택
→ 그 점을 향해 조향각 계산
→ /drive 발행
5.5 stanley_avoidance

Stanley controller와 LiDAR 기반 장애물 회피를 결합한 자율주행 코드다.

기본 흐름:

현재 위치 확인
→ raceline의 목표 waypoint 선택
→ LiDAR로 occupancy grid 생성
→ 목표점까지 충돌 여부 검사
→ 장애물이 없으면 Stanley controller
→ 장애물이 있으면 회피 목표점 생성
→ /drive 발행
5.6 state_estimation

EKF 기반 상태추정 패키지다.

보통 VESC odom과 IMU 정보를 융합해서 더 안정적인 odometry를 만들기 위해 사용한다.

5.7 safety_node

충돌 위험이 감지되면 차량을 정지시키는 안전 노드다.

일반적으로 LiDAR 기반으로 장애물까지의 거리와 현재 속도를 이용해서 충돌 시간을 계산한다.

7. 초보자가 헷갈리기 쉬운 점
/webop은 맵이 아니다

/webop은 웹 대시보드에서 보내는 주행 명령이다.

/webop = speed, steering_angle

맵은 /map이다.

RViz2는 차량을 움직이는 프로그램이 아니다

RViz2는 시각화 도구다.

RViz2 = 보기용
웹 대시보드 = 조작용
자율주행 노드 = 판단용
VESC = 실제 구동용
SLAM과 Localization은 다르다
SLAM
→ 맵을 새로 만드는 것

Localization
→ 이미 있는 맵에서 내 위치를 찾는 것
8. 가장 중요한 안전 원칙

실차에서 /drive, /webop, /teleop을 발행하면 차량이 움직일 수 있다.

따라서 처음 테스트할 때는 반드시 다음을 지킨다.

1. 바퀴를 띄운다.
2. 저속으로 테스트한다.
3. 주변 사람을 치우고 시작한다.
4. 정지 방법을 먼저 확인한다.
5. RViz2에서 /scan과 /odom이 정상인지 확인한다.
